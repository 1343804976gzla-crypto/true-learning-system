# True Learning System 设计理念与数据库治理原则

状态：执行前原则文档
版本：v1
日期：2026-03-20

## 1. 文档定位

这不是“技术方案说明书”，而是后续所有数据库拆分、日志设计、排查程序设计、恢复方案设计都必须遵守的原则文件。

如果后续某个实现与这份原则冲突，应优先修改实现，而不是修改原则。

## 2. 核心世界观

本系统不是一个页面集合，而是一个长期演化的学习系统。

它的核心不是页面，而是事实：

- 用户上传过什么
- 用户学过什么
- 用户答过什么
- 用户错过什么
- 用户为什么错
- 用户什么时候应该复习
- Agent 做过什么动作

数据库的职责是忠实保存这些事实，并让事实之间的业务边界清晰可追踪。

## 3. 原则一：业务边界优先于页面边界

不要按“前端页面长什么样”来设计数据库，要按“业务逻辑的本质是什么”来划分边界。

错误做法：

- 因为 `/wrong-answers` 页面里有挑战题和融合题，就把所有和这个页面有关的数据视为同一个模糊模块。

正确做法：

- 错题复习域只关心错题和复习闭环。
- 挑战、融合、每日复习如果本质上属于复习闭环，就归复习域。
- Agent 只拥有自己的会话和动作历史。

判断标准：

- 这个状态由谁定义。
- 这个状态由谁解释。
- 这个状态由谁承担演化成本。

## 3.1 运行时补充：写热点、增长热点、分析热点默认分离

更准确的表达应该是：

- 写热点数据、增长热点数据、长查询分析数据，默认应该隔离。
- 除非有明确证据证明混放不会带来锁竞争、备份复杂度和维护成本，否则不要混放。

优先级顺序：

1. 先按业务边界划分
2. 再按运行时热度决定物理隔离方式

也就是说：

- 业务边界先于热度分离
- 热度分离服务于业务边界

## 4. 原则二：一份事实只能有一个最终拥有者

一条业务事实只能有一个最终解释者。

例如：

- 章节真相只能由内容知识域解释
- 会话真相只能由做题运行态解释
- 错题复习状态只能由错题复习域解释
- Agent 会话状态只能由 Agent 域解释

允许：

- 其他域保存引用
- 其他域保存快照

不允许：

- 多个域同时把同一状态作为真相维护

## 5. 原则三：原始事实优先于派生结果

原始事实必须尽量完整保留。

例如：

- 上传原文要保留
- 每一题的作答明细要保留
- 每次错题重做要保留
- Agent 消息与动作要保留

派生结果可以重算：

- 每日统计
- 看板趋势
- 周报
- 聚合快照

因此：

- 原始事实库必须严肃备份
- 聚合层允许重建

## 6. 原则四：追加日志优先于覆写状态

系统越需要排查、回溯、长期维护，就越不能只保留“当前状态”。

必须同时保留：

- 当前状态
- 状态变化历史

例如：

- `wrong_answers_v2` 是当前错题状态
- `wrong_answer_retries` 是状态变化历史
- `agent_sessions` 是当前会话状态
- `agent_messages`、`agent_action_logs`、`agent_task_events` 是过程历史

## 7. 原则五：所有关键写操作必须可追踪

所有关键写操作都必须能追踪到：

- 谁触发的
- 何时触发的
- 通过哪个请求触发的
- 修改了哪个实体
- 修改前后是什么

最小追踪字段：

- `actor_key`
- `user_id`
- `device_id`
- `request_id`
- `trace_id`
- `created_at`

## 8. 原则六：跨域读取可以放宽，跨域写入必须严格

跨域读取是现实需要，但跨域写入必须极度保守。

正确做法：

- 通过应用服务编排
- 通过事件触发
- 通过专门动作接口完成

错误做法：

- 任意 Router 直接同时写多个域的核心表
- 任意脚本直接跨域批量改状态

## 9. 原则七：统计页不是主业务域

首页、历史页、学习轨迹页、看板页非常重要，但它们不是主业务真相。

它们应该被视为：

- 聚合层
- 查询层
- 读模型

约束：

- 统计与看板查询默认不应长期直接压在运行态主写库上。
- 读模型失败允许重建，但运行态主库被拖慢会直接影响用户做题和写入。

## 10. 原则八：Agent 是独立业务域，不是万能管理员

Agent 可以帮助编排业务动作，但 Agent 不应该成为“绕过业务边界的超级入口”。

Agent 正确角色：

- 会话系统
- 辅助决策系统
- 任务编排系统
- 审批与执行记录系统

Agent 错误角色：

- 直接跨域随意改所有主表
- 成为排除架构约束的后门

因此后续拆库时，Agent 必须首先独立。

## 10.1 标识补充：业务身份优先于数据库身份

实体编号不应编码“物理数据库文件名”，而应编码“业务实体类型”。

正确做法：

- 在 ID 中保留业务类型前缀
- 不在 ID 中固化物理库归属
- 通过 `trace_id` 解决跨库链路追踪

例如：

- `wa_...`
- `sess_...`
- `ags_...`

## 10.2 主实体 ID 与链路 ID 必须分离

- `public_id` 解决实体身份
- `trace_id` 解决一次跨域行为的链路追踪

它们不能混为一谈。

## 10.3 编号策略：语义自然键 + 前缀 ULID + trace_id

本项目后续统一采用三类标识：

1. 语义自然键
2. 业务实体公共 ID：`前缀 + ULID`
3. 链路追踪 ID：`trace_id`

## 10.4 迁移约束：内部行键可以保留，外部身份逐步迁移

更稳妥的做法是：

- 内部继续保留整数主键作为行键
- 新增 `public_id TEXT UNIQUE`
- 新 API、日志、跨库关联、排查工具逐步转向 `public_id`

## 11. 原则九：日志、备份、恢复不是运维附属品，而是架构本体

每次架构修改都必须同时回答：

- 怎么备份
- 怎么恢复
- 怎么检查
- 怎么审计

如果这四个问题答不上来，架构就还没有完成。

## 12. 原则十：可诊断性优先于短期开发便利

后续决策时优先级应是：

1. 能否定位
2. 能否恢复
3. 能否审计
4. 能否维护
5. 才是能否更快写完

## 13. 原则十一：数据删除默认保守

除非是明确的清理策略或明确的人为确认，否则不应轻易硬删除用户学习事实。

优先级：

1. 标记归档
2. 标记无效
3. 移入隔离区
4. 最后才是物理删除

## 14. 原则十二：迁移必须显式、可回滚、可验证

未来任何数据库拆分或迁移，都必须满足：

- 显式迁移脚本
- 迁移前快照
- 迁移后校验报告
- 回滚路径

## 15. 本项目的边界定义

内容知识底座负责：

- 上传内容
- 章节
- 知识点
- 笔记与来源

做题运行态负责：

- 学习会话
- 行为日志
- 单题明细
- 考试中间态

错题复习闭环负责：

- 错题
- 重做记录
- 每日复习卷
- 章节复习任务
- 挑战、变式、融合相关状态

Agent 负责：

- Agent 会话
- Agent 消息
- Agent 动作与任务

统计与报表负责：

- 聚合
- 查询
- 展示

不负责拥有主业务事实。

## 16. 后续数据库拆分的行为准则

1. 先定义边界，再移动表。
2. 先定义单写者，再开放读取。
3. 先加审计，再扩大写入。
4. 先做备份，再做迁移。
5. 先做校验脚本，再做批量修改。
6. 先拆独立域，再拆高耦合域。

推荐顺序：

1. Agent
2. 错题复习闭环
3. 做题运行态
4. 内容知识底座
5. 统计读模型

## 17. 日常排查哲学

不要先猜页面，要先问：

1. 这条数据属于哪个业务域。
2. 哪个域拥有它的最终写权。
3. 最近谁改过它。
4. 是否存在测试数据混入。
5. 是否有快照可回退。

## 18. 最终价值判断

最重要的不是“数据库有没有拆开”，而是：

- 业务边界清不清楚
- 关键事实会不会丢
- 问题能不能定位
- 操作能不能回溯
- 系统能不能长期维护

数据库拆分只是服务这些目标的手段。

## 19. 最终原则总结

后续一切数据库工作，以这 9 句话为总纲：

1. 页面不是边界，业务才是边界。
2. 一份事实只能有一个最终拥有者。
3. 原始事实优先于派生结果。
4. 追加历史优先于覆写状态。
5. 关键写入必须可追踪。
6. 跨域读取可以有，跨域写入必须严控。
7. 实体身份与链路身份必须分离。
8. 统计是读模型，不是主业务真相。
9. 备份、恢复、排查能力属于架构本体。
